8 /* ---------------- Original source documentation by Maurice J Fox -------
9  Debug, a class for debugging :
10 
11  This package is inspired by the similar DBUG package developed
12  for C and UNIX by Fred Fish, but does not use any of its code.
13 
14  It provides for tracing, debugging, interval timing, and output
15  control that can be controlled at run time through option strings.
16  Additionally, it provides optional function timing in connection
17  with tracing.
18 
19  Tracing:
20 
21  Tracing is printing a line of data at entry to a function and at
22  exit from the function. To get tracing, simply construct a Debug
23  object at or near entry to the function. Pass the name of the
24  as an argument string to the constructor. Tracing output will be
25  printed if tracing is enabled and either of two conditions is true:
26  The set of remembered function names is empty, or the string supplied
27  matches a remembered function name supplied as a keyword to the T:
28  option.
29 
30  Having tracing on causes tracing and debugging output indented by
31  two spaces for each level of function call up to a fixed limit,
32  currently 12. Upon execution of the object constructor, a line of
33  data goes to the output, looking something like this:
34  => fred
35  When you reach a return statement in the function, or fall
36  off the end of the code, the destructor for the object just
37  created runs, and inserts a line of data into the output,
38  looking something like this:
39  <= fred
40  So, no special effort is needed for exit tracing. The package
41  also takes care of tracking the level of function calls.
42 
43  Debugging:
44 
45  Debugging is printing a line of data to the specified output
46  destination. To get debugging, use the overloaded Dbg member
47  function or the Dbx member function on a previously constructed Debug
48  object. See the function prototypes below for the available choices.
49  In each variant of Dbg, the first argument is treated as the keyword
50  that controls debugging. The line that is printed contains indentation
51  (if tracing is in effect), the keyword followed by a colon, then the
52  requested output. The line is printed if debugging is enabled and either
53  of two conditions is true: The set of remembered keywords is empty, or
54  the keyword supplied matches a remembered keyword supplied through
55  the D: option.
56 
57  The package provides a small set of generally useful Dbg functions. If
58  these don't meet your needs, you can use the given functions as a
59  pattern to write your own. Another alternative is to use a single
60  output stream, as discussed below.
61 
62  Interval timing:
63 
64  Interval timing allows you to measure the elapsed time between
65  any two points in the program. The functions TimeStart and
66  TimeEnd are non-member functions, which means that you need
67  not create a Debug object to invoke them. (You may invoke them
68  through a Debug object if you like, however.) It also means that
69  the start and end points of the time interval(s) being measured
70  need not be in the same function. Interval timing is controlled
71  by keywords, much the same fashion as tracing or debugging.
72  Each keyword is timed independently, so the intervals being
73  timed can overlap.
74 
75  You can use one TimeStart call to mark a single point, then a
76  number of TimeEnd calls with the same keyword to get a series
77  of elapsed times from that single start point. If you call
78  TimeEnd without a preceding call to TimeStart with the same
79  keyword you will get the same result as if you had called
80  TimeStart immediately followed by TimeEnd. This won't break
81  anything, and might even be useful.
82 
83  Function Timing:
84  This is an optional capability associated with tracing. You select
85  this at compile time by the choice of constructor used for the Debug
86  object at the beginning of the function. If you use the one with
87  only one parameter, you don't get timing. If you use the one with
88  two parameters you get timing. The second parameter is a bool,
89  which you can think of as "raw timing." If it is true, you get the
90  time since the Epoch (or whatever your OS uses) on entry, and the
91  elapsed time of the function on exit. This time is purportedly in
92  seconds and milliseconds, but its precision depends upon the clock
93  tick rate on your system. If the second parameter to the constructor
94  is false, you get user friendly time in the style of asctime(),
95  which is what we use.
96 
97  So: If you use Debug D("fred"); you get
98  => fred
99  <= fred
100 
101  If you use Debug D("fred", true); you get
102  => fred 901238947.234
103  <= fred 0.877
104 
105  If you use Debug D("fred", false); you get
106  => fred Sat May 13 15:39:23 2000
107  <= fred Sat May 13 15:39:24 2000
108 
109  Option strings:
110 
111  Option strings allow run-time control of the behavior of the
112  package. If you want to see any output, you need at least one
113  option string, because the run-time default behavior is to do
114  nothing at all.
115 
116  An option string is a sequence of non-blank characters of the
117  form [anything]Option[Option . . .].
118  Option is a sequence of the form X:[keyword_list]
119  X here represents a non-blank character other than comma or colon.
120  keyword_list represents an optional sequence of "words" separated
121  by commas. Each "word" is a sequence of non-blank characters
122  other than comma or colon. The first option can be preceded by
123  a sequence of non-blank characters not containing a colon, which
124  the package ignores. The reason for doing this is given
125  under "Passing option strings to the package."
126 
127  For example, the following is a syntactically valid option string:
128  D:fred,wilma,logT:O:debug.out
129 
130  So is this one:
131  DEBUGD:fred,wilma,logT:O:debug.out
132 
133  The options are D: T: and O: D: is followed by three keywords:
134  fred wilma and log. T: is followed by no keywords, and O: is
135  followed by one keyword: debug.out
136 
137  Options currently recognized:
138 
139  T: Enables tracing. Keywords that follow this option are
140  saved and matched against the function names provided to object
141  constructors. If a match is found, the function is traced at
142  entry and exit. If no keywords are in effect, ALL functions are
143  traced at entry and exit. If you give this option with keywords
144  several times, the keywords specified are added to the list of
145  keywords recognized.
146 
147  t: Disables tracing. This causes tracing to be turned off. Also
148  the list of remembered keywords is cleared. Any keywords that
149  follow this option are ignored, but cause no error. Since the
150  option string is scanned from left to right, you could give the
151  string T:main,f1 at one point in the program. Only the
152  functions main and f1 would be traced. Later on, if you gave
153  the string t:T:, it would first have the effect of disabling
154  tracing and clearing the keywords, then enabling tracing for
155  all functions.
156 
157  D: Enables debugging. Keywords that follow the option are saved
158  and matched against the keywords provided to calls to the member
159  function Dbg. If a match is found, the line is printed to the
160  output. If no keywords are in effect, ALL calls to Dbg result in
161  lines being printed. As with T:, giving this option several times
162  with keywords causes the keywords to accumulate.
163 
164  d: Disables debugging. This causes debugging to be turned off,
165  and clears the list of remembered keywords. The same logic described
166  above for t: applies, only to debugging.
167 
168  M: Enables interval timing. Keywords that follow the option
169  are saved and matched against the keywords provided to calls
170  to the member functions TimeStart and TimeEnd. If a match is
171  found, a line is printed to the output. The function TimeStart
172  starts timing the keyword and prints the raw time. The function
173  TimeEnd stops timing the keyword and prints the elapsed time
174  since the last call to TimeStart for that keyword. If you call
175  TimeStart for the same keyword several times without intervening
176  calls to TimeEnd, the package uses the latest one called as the
177  start of the timing interval. If no keywords are in effect, ALL
178  calls to TimeStart and TimeEnd result in lines being printed.
179  As with T:, giving this option several times with keywords causes
180  the keywords to accumulate.
181 
182  m: Disables interval timing. This causes debugging to be turned off,
183  and clears the list of remembered keywords. The same logic described
184  above for t: applies, only to interval timing.
185 
186  O: Output control. If there is no keyword following, this option
187  has no effect. A keyword that follows is treated as an output
188  specifier, as follows:
189  - Specifies standard output (cout)
190  -- Specifies buffered standard error (clog)
191  Anything else is used as the filename for an open call. The
192  package does not report the success or failure of open calls, but
193  hey, it's a debugging tool, not production software. It opens the
194  file for appending, so you could switch from one file to another and
195  back again without losing data. This capability also allows you
196  to do such things as limit the size of log files and the like. If
197  you specify several keywords (useless but harmless), only the last
198  one seen actually has any effect.
199 
200  It is not necessary to supply an O: option to get output. By default,
201  the package is completely inactive, but its output is preset to clog.
202  If you just choose one of D: T: or M: you will get output to clog.
203 
204  Passing option strings to the package:
205 
206  Debug gets its option strings through the non-member function
207  Debug::specify(), which takes as an argument a pointer to a
208  C-style string. The use envisioned is that one could put Debug
209  objects all over the place, turn off tracing and debugging through
210  the specify() function and scan the command line before creating
211  the first Debug object. It's up to you, but here's a skeleton
212  example:
213 
214  int main(int argc, char *argv[]) {
215  for(int i = 1;i < argc; i++) {
216  . . .
217  if(!strncmp("-DEBUG",argv[i],6)) Debug::specify(argv[i]);
218  . . .
219  }
220  Debug D("main");
221  . . .
222  D.Dbg("A","Value of x is ",x);
223  etc . . .
224 
225  Now, while in development, you could run the program with a line
226  like -DEBUGD:T: on the command line, which would trace and debug
227  everything. By default, output goes to buffered standard error (clog).
228  A line like -DEBUGD:A,B,CT:O:debug.out would trace everything, debug
229  only those lines with keywords A, B, or C, and put the data into the
230  file debug.out.
231 
232  Another possibility is to use the package for logging. Look at this:
233 
234  int main(int argc, char *argv[]) {
235  Debug::specify("D:logt:"); // No tracing, logging with keyword log
236  for(int i = 1;i < argc; i++) {
237  . . .
238  if(!strncmp("-DEBUG",argv[i],6)) Debug::specify(argv[i]);
239  . . .
240  }
241  Debug D("main");
242  . . .
243  D.Dbg("A","Value of x is ",x);
244  D.Dbg("log","Blah blah blah ",x);
245  etc . . .
246 
247  One user suggested using an environment variable to pass in
248  an option string. That could be useful in a GUI development or
249  other circumstance where getting data from the command line is
250  not feasible.
251 
252  You can compile a standalone test/demo program by compiling
253  Debug.cpp all alone with the symbol TESTING defined. Feed it
254  option strings on the command line and watch what happens.
255 
256  Normally, the presence of the debug code will have negligible
257  effect on the performance of the application. If you really
258  feel a need to eliminate debugging in production code, you can
259  compile a new no-op version by defining the symbol DEBUG_OFF
260  both for Debug.cpp and your own code. Usually you're just
261  kidding yourself when you do that, though.
262 
263  Concerning performance, this version no longer flushes its output
264  after each line. The endl; at the end of each output statement
265  has been replaced with "\n";. This improved the speed somewhat for
266  all output choices. The big improvement over previous versions
267  was replacing clog with cout. This made a very noticeable speed
268  improvement when writing lots of debugging data to a file and a
269  smaller improvement to the other two destinations.
270 
271  The non-member function Sync() allows you to flush the output stream
272  at will. This can be useful when several processes are writing to
273  the same file, as can happen when a process forks a child in a UNIX
274  environment. It's true that multiple processes append to the same
275  in the correct sequence, that's only true of the physical transfer
276  of the buffer to the file. The system process transfers the buffer
277  when it gets full, which might not display what you want to see.
278  So, in a UNIX environment, you may want to do something like this:
279 
280  D.Sync();
281  pid = fork();
282 
283  or
284 
285  Debug::Sync();
286  pid = fork();

288  OPTIONAL CAPABILITIES:
289 
290  Multiple keywords:
291 
292  This allows you to specify several keywords, separated by commas,
293  (no spaces!) for a Dbg, Dbx, TimeStart, or TimeEnd function call.
294  Then, if any one of the keywords is in effect, (or no keywords,
295  as usual) you get the requested effect. It could have some
296  performance impact if you use a lot of keywords, so you select this
297  when compiling the package. Define the symbol DEBUG_MULT_KWDS when
298  compiling Debug.cpp to get this behavior. So, if you use this capability
299  you can write something like this:
300 
301  D.Dbg("fork,cache,file","Value of x is",x);
302 
303  and get debugging output when any of the keywords fork, cache, or file
304  has been specified, or when no keyword has been specified for
305  debugging. If you write such a line when using the version that does
306  not recognize multiple keywords, you won't get any output if any
307  keyword is specified, because the code that parses option strings
308  will not create a keyword with embedded commas, so that "keyword"
309  would never be matched. (But you would have figured that out for
310  yourself!)
311 
312  Don't get cute and write multiple keywords with consecutive commas,
313  leading commas, or trailing commas. You'll probably crash your
314  program or get some other undesired behavior. (Undesired, unless
315  you get a kick out of seeing things blow up.)
316 
317  DOS Disk Drives
318 
319  If you want to use Debug in a DOS-like environment and need to name
320  a disk drive for an output file, you have a problem because the colon
321  used to identify options clashes with the use of the colon in the
322  name of a disk drive. To make a version that uses a semicolon to
323  identify options instead of a colon, compile with the symbol
324  DEBUG_DOS_FORMAT defined. All of the foregoing documentation still
325  applies, except that where it reads colon, mentally substitute
326  semicolon. Then you can use an option string like
327 
328  D;T;O;D:\test\debug.out
329 
330  and get the output to the DOS file D:\test\debug.out. This would not
331  work in a UNIX environment, where semicolons separate commands.
332 
333  Single Output Stream.
334 
335  You can send all output through the same stream if you wish. To
336  do this, compile with the symbol DEBUG_1STREAM defined.
337  Internally, this works by logically sending all output to the clog
338  stream, changing the actual destination by using the rdbuf() member
339  function. Consequently, any clog calls made directly within your
340  program will go wherever you have sent the output.
341 
342  Why would I want to do this? you ask. The main reason is to put
343  arbitrary data into the debugging output stream without writing your
344  own Dbg calls. You can write output to clog with a line like
345 
346  clog << "Checkpoint" << n << "reached, X is now " << x <<
347  " Y is now " << Y << " Z is now " << z << endl;
348 
349  and it will go wherever you have selected using the O: option.
350  This could be a lot simpler than writing your own overloaded Dbg
351  function as suggested above.
352 
353  Because output now goes through clog, you can write something else
354  to cerr, and it will go to unbuffered standard error, not to the
355  debugging output. Of course, if you have chosen to have output
356  go to clog, they will both go to the same destination. If you do
357  not use this option, whatever you write to clog or cerr will go to
358  the same destination, intermingled with debugging output only if
359  you have specified O:--.
360 
361  Except for the documented effects of these three options, they are
362  all completely independent. This has been extensively tested.
363 */
